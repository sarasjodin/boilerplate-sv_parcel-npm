Boilerplate

En webbplats boilerplate är en färdig kodstruktur som fungerar som en startmall för att snabbt bygga en webbplats eller webbaserad applikation.

En boilerplate innehåller ofta grundläggande filer och konfigurationer, vilket gör det enklare att upprätthålla en enhetlig kodstruktur och minska uppstartstiden för nya projekt.

  • HTML-struktur (t.ex. en basmall med <head> och <body>)
  • CSS-reset eller grundläggande stilar (t.ex. Normalizer eller en grundläggande grid-layout)
  • JavaScript-konfigurationer (t.ex. en fil för interaktivitet eller en inställd JavaScript-ramverk)
  • Byggverktyg (som Parcel Webpack eller Vite för att hantera beroenden och optimera resurser)
  • Mappstruktur för att organisera koden effektivt

Ett exempel är HTML5 Boilerplate, som innehåller en optimerad grund för att snabbt starta en modern, responsiv och prestandavänlig webbplats. Jag hade tidigare aldrig hört talas om HTML5 Boilerplate, så det blir till att testa i nästa steg!

Versionshantering

För att säkerställa en strukturerad och effektiv utvecklingsprocess används Git och GitHub för versionshantering enligt bästa praxis. Detta möjliggör spårbarhet, samarbete och säker hantering av kodändringar, vilket bidrar till en smidigare arbetsflöde och en stabil kodbas.

Git-struktur

Projektet följer en grenstrategi för att underlätta utveckling och kodhantering

  • Main branch Den stabila produktionsversionen av webbplatsen
  • Dev branch Här sker aktiv utveckling innan förändringar slås samman till main.
  • Feature branches För specifika funktioner, exempelvis feature-image-optimization.

Förberedelser för att skapa en Parcel boilerplate

  1. Verifiera om Node.js LTS finns installerad på datorn och kontrollera versionen.
    • Öppna en kommandoprompt (Terminal, CMD eller PowerShell) och ange följande kommando node --version
    • Om Node.js är installerad kommer versionen att visas. Aktuell version för Node.js LTS januari 2025 är Node.js v23.7.0.
  2. Verifiera vilken Node.js-version som krävs för Parcel genom att besöka Parcels officiella webbplats.
    • Jämför med den version du har installerad.
    • Om din version är för gammal kan du behöva uppgradera.
    • Om en uppgradering behövs, se till att uppdatera Node.js på samma sätt som den ursprungligen installerades

      Det vill säga om Node.js installerades manuellt via Node.js officiella webbplats, ladda ner och installera den senaste versionen därifrån.

      Alterantivt om Node.js installerades via en pakethanterare som nvm (Node Version Manager), använd nvm för att uppdatera
      nvm install latest (för att installera den senaste versionen)
      nvm use latest (för att använda den senaste versionen)

  3. Kontrollera att npm (Node Package Manager) är installerat, vilket bör sker automatiskt med Node.js.
    • Kör följande kommando i terminalen för att verifiera versionen npm --version
  4. Skapa en ny projektmapp och navigera till den i terminalen
    • mkdir mitt-projekt && cd mitt-projekt
  5. Om du inte redan har en boilerplate för projektet, initiera ett nytt Node.js-projekt genom att köra:
    • npm init -y

    Detta skapar en package.json-fil, som behövs för att hantera beroenden och skript.

  6. Installera Parcel som ett utvecklingsberoende:
    • npm install -D parcel (Kortare nyare kod)
    • npm install --save-dev parcel (Äldre syntax men gör samma sak)

    Parcel är en snabb och enkel bundler som används för att hantera och bygga ditt projekt.

Om du redan har en färdig boilerplate, gå vidare till rätt avsnitt i dokumentationen.

Arbetsflöde

Efter att du har satt upp din Parcel boilerplate, kan du nu börja utveckla din webbplats med ett effektivt arbetsflöde. Parcel är en zero-config bundler, vilket betyder att du kan börja direkt utan att behöva konfigurera något.

  1. Klona repot

    • När du skapar ett nytt projekt baserat på en boilerplate, behöver du klona repot från GitHub (eller en annan versionshanteringstjänst) för att få en kopia av den befintliga kodbasen.

    • git clone <repo-url>
    • cd <repo-namn>
  2. Skapa en ny branch

    • När du jobbar i ett Git-projekt är det viktigt att skapa en ny branch för att separera din utveckling från den stabila koden.

    • git checkout -b feature-newpage
  3. Arbeta och göra commits

    • När du arbetar med kod i ett Git-projekt måste du lägga till ändringarna samt spara och spåra ändringarna. Detta görs genom add och commit (lägga till och spara en version av koden).

      git add .
    • git commit -m "Lagt till ny undersida för processbeskrivning"
  4. Pusha förändringar

    • När du har gjort ändringar i din kod och skapat en commit, finns dessa ändringar bara på din lokala dator. För att dela dem med andra och spara dem i det fjärrlagrade (remote) repositoryt på GitHub eller en annan server, behöver du pusha ändringarna.

      git push origin feature-newpage
  5. Skapa en pull request och granska innan merge till dev

    • När du har pushat dina ändringar till GitHub, behöver du skapa en Pull Request (PR) för att be om att få dina ändringar granskade och sammanslagna (merged) till dev-branchen.

    Skapa en PR via terminalen (GitHub CLI krävs)

    • gh pr create --base dev --head feature-newpage --title "Lagt till ny undersida" --body "Denna PR lägger till en ny undersida."
  6. Testa utvecklingen

    Att testa i dev-läge innebär att köra projektet lokalt, felsöka, testa UI och köra kodvalidering innan koden går vidare till main.

    Exempel på verifieringar och kommandon som kan användas vid testning

    Starta utvecklingsservern lokalt med Parcel npm start

    Testa funktioner och UI

    • Klicka runt och se att allt fungerar som förväntat.
    • Kontrollera konsolen (F12 → Console) för fel.
    • Testa i olika webbläsare och enheter.

    Verifiera kodkvalitet med ESLint och Prettier npm run lint och npm run format

    Gör en manuell build för att testa produktionsläge npm run build och npx serve dist

  7. När utvecklingen är stabil, slå samman dev till main och publicera automatiskt

    När alla funktioner är testade, vill vi slå samman (merge) dev till main och publicera de färdiga ändringarna live.

    Hämta senaste ändringar från main

    • git checkout main && git pull origin main

    Slå samman dev till main

    • git merge dev --no-ff -m "Slår samman dev till main efter godkänd PR"

    Pusha till main

    • git push origin main

    En .gitignore-fil används för att exkludera filer som node_modules/, dist/, och andra byggfiler från versionshanteringen.

  8. Kör bygg- och deploy-process efter merge

    Hämta senaste ändringar i main

    • git pull origin main

    Installera beroenden om nödvändigt

    • npm install

    Bygg projektet

    • npm run build

    Deploya till GitHub Pages

    • npx gh-pages -d dist

    För Netlify/Vercel

    • För Netlify har jag skapat en netlify.toml-fil som hjälper till med inställningarna för deply-funktionaliteten. >
    • vercel --prod

Automatisering med Parcel

Parcel används för att automatisera uppgifter som minifiering, transpilering och optimering.

Minifiering av CSS

Parcel hanterar automatiskt minifiering av CSS i produktionsläge.

  • Kommando npm run build
  • Utdata Minifierade CSS-filer sparas i dist/-mappen
  • Fördelar Snabbare laddningstid och minskad filstorlek

Transpilering och minifiering av JavaScript

För att säkerställa kompatibilitet transpileras JavaScript med Babel.

  • Parcel konfigurerar automatiskt Babel.
  • Kommando npm run build
  • Utdata JavaScript-filer i dist/-mappen, optimerade och ES5-kompatibla
  • Fördelar Bakåtkompatibilitet och prestandaförbättring

Minifiering av HTML

HTML-filer optimeras för att reducera storlek

  • Kommando npm run build
  • Utdata Komprimerade HTML-filer i dist/
  • Fördelar Snabbare laddning och mindre bandbreddskonsumtion

Optimering av bilder

Optimering skall ske automatiskt vid uppdatering av bilder

Automatisk bildoptimering genom modernare format som WebP och AVIF

  • Tillägg Paket som imagemin kan användas
  • Kommando npm run build
  • Utdata Optimerade bilder i dist/images/
  • Fördelar Minskad filstorlek, snabbare laddning

I mitt boilerplate-projekt använde jag imagemin för att optimera bilder. Dock stötte jag på flera felmeddelanden relaterade till beroenden som behövde uppdateras. För att undvika kompatibilitetsproblem valde jag att ta bort imagemin efter att WebP-filerna skapats. Nästa steg är att utvärdera andra bildoptimeringsverktyg och justera inställningarna för att säkerställa att bilderna behåller en bra balans mellan kvalitet och filstorlek.

Bakåtkompabilitet

  • Bakåtkompabilitet i genererade filer, så att exempelvis JavaScript transpileras så att dessa fungerar i så många webbläsare som möjligt.
  • Kopiering av arbetsfiler till mapp för publicering med optimerade filer.
  • För att säkerställa kompatibilitet med äldre webbläsare används Babel och polyfills. Parcel hanterar detta automatiskt genom sin bundler.
  • JavaScript transpileras till ES5.
  • CSS autoprefixas för äldre webbläsare.

Kopiering av arbetsfiler till mapp för publicering med optimerade filer

Vid byggprocessen kopieras alla optimerade filer till en dist/-mapp för publicering.

  • Kommando npm run build
  • Utdata Publiceringsfärdiga filer i dist/

Kör npm start

Startar Parcel i utvecklingsläge.

  • Laddar filer från src/ och har hot-reload (automatiska uppdateringar när du ändrar kod).
  • Använder en inbyggd utvecklingsserver (ofta på 🔓 http://localhost:1234/).

Kör npx serve dist

Serverar den byggda versionen från dist/, alltså den optimerade produktionskoden.

  • Ingen hot-reload – du måste köra npm run build manuellt om du gör ändringar.
  • Använder ofta 🔓 http://localhost:3000/ (beroende på vad serve väljer).

Netlify-specifik aktivitet

Netlify spelar en viktig roll i automatiseringen och publiceringen av ditt projekt. Här är en detaljerad genomgång av vad Netlify-kontot och appen gör i varje steg av arbetsflödet

  • Automatisk övervakning

    Netlify övervakar ditt repo. Varje gång du pushar till en viss branch (t.ex. main eller build), triggas en ny automatisk deployment. Du kan ställa in vilken branch som ska byggas och publiceras.

  • Bygger projektet automatiskt

    Netlify kör kommandot du angett för att bygga projektet och skapar en optimerad version av webbplatsen i en dist/-mapp. Netlify lagrar denna version och gör den redo att publiceras.

  • Publicerar webbplatsen live

    Efter att bygget är klart, publicerar Netlify webbplatsen automatiskt. Netlify kopierar den färdigbyggda webbplatsen till sin CDN (Content Delivery Network). Webbplatsen blir tillgänglig på en Netlify-domän men du kan även koppla en egen domän.

  • Hanterar miljövariabler och konfiguration

    Om du behöver API-nycklar eller andra känsliga data kan du spara dem som miljövariabler i Netlify istället för att ha dem i koden.

  • Kan skapa preview deployments

    Om du jobbar på en feature-branch och vill testa ändringar innan de går live, skapar Netlify en Deploy Preview.

    Varje Pull Request (PR) får en egen tillfällig länk där du kan testa ändringarna.

    Du kan dela denna länk med teamet för att få feedback innan merge till main.

  • Hanterar formulär och serverfunktioner

    Netlify har inbyggt stöd för formulärhantering och serverlösa funktioner utan behov av backend.

    Ett HTML-formulär kan fånga data direkt med Netlify Forms och JavaScript-funktioner kan köras på servern med Netlify Functions.
